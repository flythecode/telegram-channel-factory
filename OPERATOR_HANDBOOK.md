# Telegram Channel Factory — Operator Handbook

Короткая памятка для человека, который поддерживает продукт после запуска.

## 1. Что должен уметь оператор

Оператор отвечает не за разработку, а за то, чтобы продукт оставался в рабочем состоянии:
- проверить, что API, worker и bot живы
- понять, это локальный сбой, инфраструктурный сбой или проблема конкретного пользователя
- безопасно перезапустить сервисы
- быстро пройти smoke path после изменений
- собрать понятный фактологический отчёт, если нужна эскалация разработчику

---

## 2. Где смотреть правду

Главные источники состояния:
- `README.md` — как вообще устроен продукт
- `RELEASE_SMOKE.md` — короткий обязательный smoke path
- `FINAL_QA_RUNBOOK.md` — полный ручной пользовательский сценарий
- `docker-compose.yml` — какие сервисы должны быть подняты
- `.env` / production env file — в каком runtime-mode запущен стенд
- audit history в продукте — что именно делал пользователь

Если данные расходятся, источником истины считается реальное состояние runtime + audit trail.

---

## 3. Нормальное состояние системы

В норме должно быть так:
- API отвечает `200` на `/health`
- bot runtime не crash-loop'ится
- worker поднят и продолжает обрабатывать очередь
- публикации получают понятные статусы (`queued`, `sent`, `failed`)
- пользователь может заново открыть канал, зайти в настройки, черновики и публикации

Минимальная проверка:

```bash
curl http://127.0.0.1:8000/health
```

Если есть compose-окружение:

```bash
docker compose ps
docker compose logs --tail=100 api worker bot
```

---

## 4. Ежедневный короткий check

Раз в день или после любого деплоя:

1. Проверить `GET /health`
2. Проверить, что `api`, `worker`, `bot` в healthy/running состоянии
3. Проверить, что нет свежей серии `failed` публикаций без понятной причины
4. Проверить, что новые черновики/публикации продолжают проходить pipeline
5. Если был релиз — прогнать `RELEASE_SMOKE.md`

---

## 5. Самые частые инциденты и что делать

### A. Бот не отвечает на `/start`
Проверить:
- жив ли контейнер/процесс `bot`
- не сломан ли `TELEGRAM_BOT_TOKEN_FILE` / env
- нет ли exception в bot logs

Действие:
- снять последние логи
- перезапустить только `bot`
- снова проверить `/start`

### B. API не отвечает /health
Проверить:
- жив ли `api`
- применились ли миграции
- доступна ли база
- нет ли ошибки конфигурации после релиза

Действие:
- снять логи `api`
- проверить DB connectivity
- если причина в релизе — откатить/починить release flow до повторного smoke

### C. Публикации застряли в `queued`
Проверить:
- жив ли `worker`
- есть ли ошибки Telegram publisher / stub publisher
- не упёрся ли runtime в retry loop

Действие:
- снять `worker` logs
- подтвердить, что очередь снова двигается после restart
- если сбой внешний (Telegram/API) — зафиксировать это в отчёте

### D. Пользователь говорит, что «ничего не происходит»
Проверить:
- audit history по проекту/сущности
- какой был последний action
- создался ли draft / publication / task
- не нажал ли пользователь действие повторно

Действие:
- опираться на audit trail, а не на догадки
- зафиксировать entity id, статус, время, последний action

---

## 6. Безопасный restart

Если нужен restart после инцидента:

```bash
docker compose ps
docker compose logs --tail=100 api worker bot
docker compose restart api worker bot
curl http://127.0.0.1:8000/health
```

Что важно:
- сначала посмотреть логи, потом перезапускать
- не делать несколько хаотичных restart подряд
- после restart всегда подтверждать `/health` и базовый bot/user path

---

## 7. Обязательный post-deploy smoke

После любого релиза или hotfix:

```bash
python3 -m compileall app scripts
pytest -q tests/test_e2e_mvp_flow.py tests/test_channel_connection_flow.py tests/test_bot_navigation.py tests/test_worker.py tests/test_runtime_hardening.py tests/test_telegram_publisher.py
```

Если есть deploy-среда:

```bash
curl http://127.0.0.1:8000/health
```

Релиз нельзя считать успешным, пока не подтверждены:
- health ok
- bot отвечает
- queue path жив
- нет нового критического regressions

---

## 8. Когда эскалировать разработчику сразу

Не чинить «на глаз», а сразу эскалировать, если:
- миграции ломают запуск
- audit trail противоречит фактическому состоянию данных
- есть повторяемая потеря draft/publication/state
- Telegram publisher даёт системные ошибки после валидного restart
- после релиза не проходит основной path `/start -> setup -> draft -> publication`

В эскалации должны быть:
- время инцидента
- environment
- какой пользователь/проект затронут
- entity ids / request id
- последние логи `api` / `worker` / `bot`
- что уже проверили и что не помогло

---

## 9. Что оператор не должен делать

Оператору нельзя:
- править production-данные вручную без понимания последствий
- менять токены/секреты без отдельного разрешения
- пропускать smoke-check после релиза
- объявлять «всё работает», если не подтверждён пользовательский путь
- маскировать проблему restart'ами без логов и отчёта

---

## 11. Definition of done для оператора

Инцидент считается закрытым, если:
- причина хотя бы локализована
- сервисы снова в нормальном состоянии
- smoke path после действия подтверждён
- есть краткая запись, что сломалось, что сделали и что получилось
