# Telegram Channel Factory — User Flows v1

## 1. Purpose of this document

Этот документ фиксирует пользовательские сценарии продукта **Telegram Channel Factory** на уровне потока действий пользователя внутри Telegram-бота.

Цель документа:
- описать путь пользователя от первого входа до рабочего канала
- зафиксировать основные ветки поведения
- показать, где пользователь создаёт, настраивает, запускает и потом меняет канал
- дать основу для Telegram UI map, bot FSM и backend API design

---

## 2. Main product promise reflected in flows

User flows строятся вокруг 2 ключевых режимов продукта:

1. **Setup flow** — пользователь быстро создаёт и запускает новый канал-проект
2. **Ongoing management flow** — пользователь затем возвращается и управляет уже созданным каналом

Принцип: продукт не заканчивается после онбординга, а переходит в постоянный режим управления.

---

## 3. Primary user journey map

Главный путь пользователя в MVP v1 выглядит так:

1. Пользователь открывает бота и нажимает `/start`
2. Получает стартовое сообщение и главное меню
3. Выбирает **Создать канал**
4. Проходит мастер создания проекта
5. Настраивает базовые параметры канала
6. Выбирает команду ИИ-агентов
7. Подключает Telegram-канал
8. Выбирает режим работы канала
9. Запускает первый контентный цикл
10. Получает контент-план / черновики / публикации
11. Возвращается позже в **Мои каналы**
12. Меняет настройки, агентов, план или режим работы

# 5. Detailed flows

## Flow A — First start and onboarding

### Goal
Помочь пользователю понять, что умеет бот, и быстро направить его в главное действие.

### Trigger
Пользователь впервые открывает бота или вводит `/start`.

### Entry point
`/start`

### User path
1. Пользователь отправляет `/start`
2. Бот показывает короткое приветствие
3. Бот объясняет в 1–2 фразах, что он помогает создавать и управлять Telegram-каналами с ИИ-агентами
4. Бот показывает главное меню

### Main buttons
- Создать канал
- Мои каналы
- Как это работает
- Помощь

### Success outcome
Пользователь понимает, что делать дальше, и идёт либо в создание канала, либо в просмотр существующих проектов.

---

## Flow B — Create new channel project

### Goal
Создать новый проект канала через простой пошаговый сценарий.

### Trigger
Пользователь нажимает **Создать канал**.

### User path
1. Бот предлагает начать создание проекта
2. Бот спрашивает название проекта
3. Бот спрашивает тематику / нишу канала
4. Бот спрашивает язык канала
5. Бот спрашивает цель канала в упрощённой форме
6. Бот спрашивает тип контента
7. Бот спрашивает желаемую частоту публикаций
8. Бот показывает краткое summary настроек
9. Пользователь подтверждает создание проекта
10. Бот создаёт проект и переводит пользователя к следующему шагу — настройке агентной команды

### User inputs
- текстовое название проекта
- выбор ниши из кнопок или ввод своей ниши
- выбор языка
- выбор частоты публикаций
- выбор типа контента

### Main buttons
- Назад
- Пропустить
- Подтвердить
- Отменить

### Success outcome
Создан базовый проект канала с начальными настройками.

---

## Flow C — Configure AI agent team

### Goal
Дать пользователю возможность выбрать и настроить команду из 3–7 ИИ-агентов.

### Trigger
После создания проекта бот переводит пользователя в настройку агентной команды.

### User path
1. Бот объясняет, что канал будет вести команда ИИ-агентов
2. Бот предлагает preset-команды
3. Пользователь выбирает вариант: 3, 5 или 7 агентов
4. Бот показывает состав выбранной команды и краткое описание ролей
5. Пользователь подтверждает preset или нажимает изменить
6. Пользователь может:
   - выключить лишнего агента
   - включить дополнительного агента
   - изменить базовую роль агента
7. Бот сохраняет команду агентов
8. Бот переводит пользователя к подключению канала

### Possible presets
- Быстрый старт — 3 агента
- Сбалансированная команда — 5 агентов
- Расширенная редакция — 7 агентов

### Example roles
- Strategist
- Researcher
- Writer
- Editor
- Fact-checker
- Reach optimizer
- Publisher

### Main buttons
- 3 агента
- 5 агентов
- 7 агентов
- Настроить вручную
- Подтвердить команду
- Назад

### Success outcome
У проекта появляется активная агентная команда с понятным составом и ролями.

## Flow D — Connect Telegram channel

### Goal
Привязать реальный Telegram-канал к проекту и убедиться, что бот может публиковать.

### Trigger
Пользователь завершил настройку проекта и команды агентов.

### User path
1. Бот спрашивает, есть ли уже готовый Telegram-канал
2. Пользователь выбирает:
   - Да, канал уже есть
   - Нет, покажи как создать
3. Если канал уже есть:
   - бот просит указать `@username` или ID канала
   - бот даёт инструкцию добавить бота в админы
   - бот просит подтвердить выполнение шага
   - бот проверяет возможность публикации / корректность подключения
4. Если канала ещё нет:
   - бот даёт краткую инструкцию по созданию канала вручную
   - после этого возвращает пользователя в шаг подключения
5. Бот показывает статус подключения
6. Пользователь подтверждает завершение подключения

### Main buttons
- У меня уже есть канал
- Создать и подключить вручную
- Проверить подключение
- Повторить инструкцию
- Назад

### Success outcome
Канал успешно привязан к проекту, бот имеет необходимые права.

---

## Flow E — Launch content pipeline

### Goal
Запустить первый рабочий цикл генерации контента после настройки проекта.

### Trigger
Проект создан, агентная команда настроена, канал подключён.

### User path
1. Бот сообщает, что проект готов к запуску
2. Бот предлагает действия первого старта:
   - создать контент-план
   - сгенерировать первые идеи
   - сгенерировать первые черновики
3. Пользователь выбирает нужное действие
4. Бот запускает pipeline
5. Бот показывает результат генерации
6. Пользователь переходит к просмотру задач / черновиков / очереди

### Main buttons
- Создать контент-план
- Сгенерировать 10 идей
- Создать 3 черновика
- Открыть проект

### Success outcome
Пользователь получает первые рабочие материалы по каналу и переходит из setup в управление.

---

## Flow F — Manage drafts and publications

### Goal
Позволить пользователю просматривать, согласовывать и публиковать контент.

### Trigger
У проекта появились задачи, черновики или запланированные публикации.

### User path
1. Пользователь открывает канал из списка **Мои каналы**
2. Нажимает **Черновики** или **Очередь публикаций**
3. Видит список элементов
4. Открывает конкретный черновик
5. Выбирает действие:
   - approve
   - reject
   - редактировать
   - перегенерировать
   - отправить в очередь
6. Если публикация уже в очереди, пользователь может:
   - изменить время
   - запустить сейчас
   - остановить
   - удалить из очереди
7. Бот подтверждает действие и обновляет статус

### Main buttons
- Одобрить
- Отклонить
- Перегенерировать
- Редактировать
- В очередь
- Опубликовать сейчас
- Изменить время
- Назад

### Success outcome
Пользователь управляет публикациями без выхода из Telegram-бота.

### Trigger

3. Видит список каналов
4. Выбирает конкретный канал

   - Настройки
   - Агенты
   - Контент-план
   - Черновики
   - Очередь
   - Режим работы
7. Бот переводит пользователя в выбранный раздел

### Main buttons
- Настройки
- Агенты
- Контент-план
- Черновики
- Публикации
- Режим
- Назад

### Success outcome
Пользователь легко возвращается к своему каналу и может управлять им после создания.

## Flow H — Modify agent team after launch

### Goal
Позволить менять агентную команду не только на этапе первоначальной настройки.

### Trigger
Пользователь открывает экран **Агенты** внутри конкретного канала.

### User path
1. Пользователь открывает проект
2. Нажимает **Агенты**
3. Видит текущую команду и роли
4. Выбирает конкретного агента или действие команды
5. Может:
   - выключить агента
   - включить агента
   - сменить роль
   - изменить базовую конфигурацию
   - заменить preset команды
6. Бот сохраняет изменения
7. Бот подтверждает обновление конфигурации

### Main buttons
- Состав команды
- Изменить preset
- Выключить агента
- Включить агента
- Изменить роль
- Настройки агента
- Назад

### Success outcome
Пользователь гибко меняет команду агентов уже после запуска проекта.

## Flow I — Change mode / schedule / content strategy

### Goal
Дать пользователю возможность корректировать операционный режим канала после запуска.

### Trigger
Пользователь открывает раздел **Настройки**, **Режим** или **Контент-план**.

### User path
1. Пользователь открывает нужный раздел
2. Видит текущие настройки
3. Выбирает параметр для изменения
4. Меняет:
   - режим работы
   - частоту публикаций
   - контентные рубрики
   - стиль подачи
   - длину постов
   - policy согласования
5. Бот показывает краткое подтверждение изменений
6. Обновлённая конфигурация начинает применяться к следующим задачам и публикациям

### Modes
- Manual
- Semi-auto
- Auto

### Main buttons
- Manual
- Semi-auto
- Auto
- Изменить расписание
- Изменить рубрики
- Изменить стиль
- Сохранить
- Назад

### Success outcome
Пользователь может менять стратегию канала в процессе эксплуатации, не пересоздавая проект.

### Entry points
- кнопка **Как это работает**

- контекстная помощь на сложных шагах

### What user can learn

## Flow K — Empty states

### Goal
Не оставлять пользователя в тупике, если данных ещё нет.

### Important empty states
- нет ни одного канала
- нет агентной команды
- нет контент-плана
- нет черновиков
- нет публикаций в очереди

### Required UX behavior
Каждый empty state должен содержать:
- короткое объяснение
- одну главную CTA-кнопку
- понятное следующее действие

---

## Flow L — Cancel / back / recovery flow

### Goal
Позволить пользователю безопасно выйти из сценария и вернуться позже.

### Required behavior
- на ключевых шагах есть кнопка **Назад**
- в setup flow пользователь может **Отменить**
- бот должен сохранять промежуточный прогресс там, где это уместно
- пользователь должен иметь возможность вернуться к незавершённому setup позже

# 8. State transitions at user level

Основные пользовательские переходы в системе:

1. **New user** → `/start`
2. `/start` → **Главное меню**
3. **Главное меню** → **Создать канал**
4. **Создать канал** → **Проект создан**
5. **Проект создан** → **Команда агентов настроена**
6. **Команда агентов настроена** → **Канал подключён**
7. **Канал подключён** → **Первый pipeline запущен**
8. **Первый pipeline запущен** → **Черновики/публикации доступны**
9. **Черновики/публикации доступны** → **Постоянное управление каналом**

---

# 9. UX constraints to keep flows healthy

1. На каждом экране должно быть не больше одного главного CTA.
2. Сложные решения надо давать через короткие пресеты, а не через перегруженные формы.
3. Пользователь должен понимать, что происходит сейчас и что будет дальше.
4. Любой важный flow должен быть обратим или редактируем.
5. После setup пользователь должен ощущать, что каналом можно управлять постоянно.

---

# 10. MVP flow completion criteria

User flows v1 считаются корректно реализованными, если:
- новый пользователь проходит путь от `/start` до рабочего проекта без ручной помощи
- пользователь может создать проект и подключить канал
- пользователь может выбрать и сохранить команду агентов
- пользователь может запустить первый контентный цикл
- пользователь может одобрять и публиковать контент
- пользователь может позже вернуться и изменить основные настройки проекта
- навигация остаётся понятной и не ломает ощущение контроля
