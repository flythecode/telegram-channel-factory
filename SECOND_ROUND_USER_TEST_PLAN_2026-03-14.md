# Telegram Channel Factory — Second Round User Test Plan (2026-03-14)

## Status

Пакет подготовлен для второго раунда реальных пользовательских тестов после UX-исправлений из backlog item 135.

**Важно:** backlog item 136 можно закрыть только после 2+ живых прохождений обновлённого сценария реальными участниками. В этой среде без внешнего контакта можно подготовить пакет и зафиксировать blocker, но нельзя честно отметить задачу выполненной.

## Goal

Подтвердить на реальных людях после внесённых UX-исправлений, что:
- onboarding теперь читается без помощи оператора
- путь `создать проект -> подключить канал -> настроить -> сгенерировать -> опубликовать` проходится самостоятельно
- ранее исправленные UX-боли действительно исчезли или заметно ослабли

## Focus of round 2

Вторая волна должна проверять не просто базовую работоспособность, а именно эффект после исправлений:
- понятность стартового экрана и главного меню
- ясность CTA между `контент-план -> идеи -> черновики -> публикация`
- понимание статусов generation / лимитов / блокировок
- способность вернуться в канал позже и продолжить работу без подсказки

## Required assets before inviting users

1. Тестовый бот отвечает на `/start`
2. Доступен `FIRST_USER_LAUNCH_PATH.md`
3. Оператор открывает этот файл и пакет `controlled-user-tests/round-2-2026-03-14/`
4. Для каждого участника заранее подготовлен отдельный participant file

## Participants

Минимум 2 реальных пользователя:
- хотя бы 1 без глубокого продуктового контекста
- желательно 1 человек, который раньше уже видел старый UX, чтобы сравнить восприятие до/после

## Moderator rules

Нельзя:
- объяснять интерфейс заранее
- подсказывать следующий шаг до явной блокировки
- выполнять скрытые developer-only действия за участника

Можно:
- напомнить цель теста
- просить проговаривать мысли вслух
- фиксировать время, паузы, вопросы и места потери контекста

## Test script

### Scenario A — Onboarding after fixes
1. Открыть бота
2. Отправить `/start`
3. Понять, с чего начать
4. Создать проект
5. Выбрать preset

Фиксировать:
- замечает ли пользователь правильную первую CTA
- понимает ли, что будет дальше
- просит ли помощь на старте

### Scenario B — Setup to generation
1. Подключить канал
2. Дойти до channel dashboard
3. Создать контент-план
4. Сгенерировать идеи
5. Создать черновики

Фиксировать:
- понимает ли пользователь последовательность шагов
- различает ли `идеи`, `черновики`, `публикации`
- замечает ли generation status / limit messaging

### Scenario C — Publish and return later
1. Открыть черновик
2. Подтвердить или отредактировать
3. Создать публикацию
4. Опубликовать сразу или поставить в очередь
5. Вернуться позже и снова открыть тот же канал

Фиксировать:
- понятна ли разница между immediate publish и queue
- может ли пользователь сам вернуться к проекту позже
- теряется ли он в статусах или навигации

## Success criteria for closing backlog item 136

Пункт 136 можно отметить `[x]`, только если выполнено всё ниже:
- минимум 2 реальных пользователя прошли вторую волну
- по каждому заполнен отдельный participant result file
- в `SECOND_ROUND_USER_TEST_REPORT_2026-03-14.md` есть реальные наблюдения по улучшениям/оставшимся болям
- есть однозначный verdict: UX-путь после исправлений стал самопроходимым или нет

## Closure rule

Без живых участников и заполненных результатов пункт 136 остаётся незавершённым.
